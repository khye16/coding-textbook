## 1. 프로젝트 정체성

**이름**: Python & C 디지털 교재  
**버전**: v1.0  
**대상 학생**: 강사의 지인 극소수 (10명 이하)  
**목표**: 8~12개월에 걸친 Python/C 입문 → 주니어 지원 가능한 기초 포트폴리오 완성

### 본질

이 프로젝트는 **앱 개발 프로젝트가 아니라 교육 콘텐츠 운영 프로젝트**다.
웹사이트는 교재를 전달하기 위한 최소 도구일 뿐이다.
기능 개발은 콘텐츠 작성 속도를 방해하지 않는 범위에서만 진행한다.

### 한 줄 정의

```
GitHub Pages에 올라간 VitePress 교재
+ 각 단원에 "Codespaces로 실행" 버튼
+ devcontainer 한 번 설정으로 학생 환경 0세팅
```

## 4. 작업 방식 (바이브 코딩 워크플로우)

### 4단계 프로세스

```
[1] Research  →  research.md 작성
[2] Plan      →  plan.md 작성
[3] Annotation Cycle  ←→ 사용자가 NOTE 추가, 반복
[4] Implementation    →  실제 파일 생성
```

### 절대 규칙

1. **"아직 구현하지 마"가 없는 메시지에서는 코드/파일을 만들지 마라.**
   - 사용자가 명시적으로 "구현해" 또는 "만들어"라고 할 때만 파일 작성.
   - 그 외에는 research.md / plan.md만 작성.

2. **모든 분석/계획은 반드시 마크다운 파일로 저장하라.**
   - 채팅창 안의 구두 요약은 금지.
   - `research.md`, `plan.md` 등 영구 파일로.

3. **"깊이(deeply)", "상세히(in great detail)" 요청이 오면 진짜 깊이 분석하라.**
   - 함수 시그니처 수준에서 멈추지 말고 세부 사항까지.
   - config 키, 명령어 옵션, 폴더 구조까지.

4. **사용자가 NOTE를 plan.md에 추가했다고 하면, 모든 NOTE를 정확히 반영하라.**
   - 한 개도 누락 금지.
   - 반영 후 plan.md를 통째로 다시 보여주지 말고, 변경된 부분만 요약.

5. **막히면 멈추고 물어봐라.**
   - 임의로 우회하거나 plan에서 벗어나지 마라.
   - 혼자 판단해서 새 기능을 추가하지 마라.

## 5. 콘텐츠 작성 규칙 (단원 작성 시)

### 단원 본문 표준 구조

```markdown
# [번호] [제목]

## 이걸 왜 배우나
1~2문장.

## 핵심 개념
본인 언어로 작성된 본문.

## 예시 1 / 2 / 3
간단한 것부터 응용까지.

## 흔한 실수
2개 이상.

## 자가진단
3문제 이상.

## 직접 풀어보기

[![Open in GitHub Codespaces](https://github.com/codespaces/badge.svg)](https://codespaces.new/USER/REPO?quickstart=1)

Codespaces가 열리면 아래 폴더로 이동하세요:
\`\`\`bash
cd exercises/python/01-hello-world
\`\`\`

연습 문제는 자체 작성한다. 외부 사이트에 의존하지 않는다.

필수 미니 과제:
- 단원 본문 흐름에 자연스럽게 풀어볼 코드 과제.
- `exercises/<lang>/<번호>-<제목>/` 폴더에 starter 파일 제공.

추가 도전 문제:
- 본문 범위 내 응용 문제.
- 같은 폴더에 추가 starter 파일 제공.
- 학생이 직접 실행하고 강사가 검수.

문제 1: ... (파일명, 실행 명령어, 예상 출력 명시)
문제 2: ...

## 참고 자료
- [자료 1](url)
- [자료 2](url)
```

### 저작권 규칙 (절대)

1. 외부 책/사이트 단락을 그대로 복사 금지. 반드시 본인 언어로 재서술.
2. 코드 예시는 직접 작성 (변수명, 구조를 외부 자료와 다르게).
3. 백준, 프로그래머스 등 외부 문제 사이트 문제를 베끼지 마라. 문제 본문, 입출력 예시, 변수명까지 직접 작성한다.
4. 단원 끝에 참고 자료 출처 명시.
5. 외부 자료 문장이 그대로 들어갔는지 확신 없으면 검색해서 확인하라.

## 6.1 자체 문제 작성 규칙

- 외부 사이트(백준, 프로그래머스 등) 문제 베끼지 마.
  아이디어 비슷해도 본문, 입출력 예시, 변수명 전부 직접 작성.
- 문제는 단원에서 다룬 개념 안에서만 풀 수 있어야 함.
  (예: `input()` 안 다룬 단원에선 input 필요한 문제 금지)
- 각 문제마다 명시 필수:
  - 문제 설명
  - 파일명
  - 실행 명령
  - 예상 출력(또는 출력 조건)
- 자동 채점 없음. 학생 직접 출력 확인 + 강사 검수.

## 6.2 Starter 파일 작성 규칙

starter 파일은 답이 거의 다 쓰여있으면 안 됨.
학생이 채울 부분만 비어있고, 그 외 구조/주석만 제공해야 함.

✅ 올바른 starter 예시:

```python
# 문제: Hello, Python! 을 출력하세요
# 힌트: print 함수와 따옴표를 사용합니다
# 여기에 코드를 작성하세요
```

❌ 잘못된 starter 예시:

```python
# Hello, Python! 을 출력하세요
print("Hello, Python!")
```

규칙:

- 답 코드 절대 미리 작성 금지
- 학생이 채울 위치는 `# 여기에 코드를 작성하세요` 주석으로 명시
- 필요시 짧은 힌트 1줄 (사용할 함수 이름 정도)
- 함수 정의가 필요한 문제는 시그니처만 제공, 본문은 `pass` 또는 비움

## 6.3 파일 명명 규칙

학생에게 본문 순서와 파일 순서가 일치하게 보이도록 숫자 접두사 사용.

폴더:

- `docs/<lang>/<번호>-<제목>.md` (예: `docs/python/01-hello-world.md`)
- `exercises/<lang>/<번호>-<제목>/` (예: `exercises/python/01-hello-world/`)

폴더 안 파일:

- `<순번>_<이름>.py` 또는 `.c`
- 예: `01_hello.py`, `02_greeting.py`, `03_badge.py`, `04_quotes.py`

본문에서 "문제 1: `01_hello.py`" 형식으로 번호 매칭이 명확하게 보이게.

## 6.4 학생용 표준 안내 박스

모든 단원의 "직접 풀어보기" 섹션에 다음 안내가 반드시 포함되어야 함:

### 1단원에만 (한 번만 가르침)

- **Codespaces 기본 조작 박스**: 파일 열기 / 코드 작성 / 저장 / 실행 / 출력 확인 5단계

### 모든 단원에 (표준)

- **초기화 안내 박스**:
  초기화하기
  문제를 다시 처음부터 풀고 싶을 때는 터미널에서:
  ```bash
  git checkout exercises/python/01-hello-world/<파일명>
  ```
  이렇게 실행하면 해당 파일이 starter 상태로 돌아갑니다.
  (단원마다 경로 부분만 바꿈)

## 8. 학생 경험 원칙

학생이 막히지 않게 단원에 반드시 포함:

1. **정확한 폴더 경로**: `cd exercises/python/01-hello-world`
2. **파일명 명시**: "`01_hello.py` 파일을 열고..."
3. **실행 명령어**: `python3 01_hello.py` 또는 `gcc 01_hello.c -o 01_hello && ./01_hello`
4. **예상 출력**: 정확히 어떤 결과가 나와야 하는지 명시

"코드 작성" 같은 모호한 지시 금지. 학생이 터미널에 뭘 쳐야 하는지 정확히 적기.
